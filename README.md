# COGNIX — Support that remembers every customer

An AI-powered customer support system where **Hindsight persistent memory** gives every agent the full conversational history of every customer — so customers never repeat themselves. Responses are generated by **Groq (Qwen 32b)** via a single structured LLM call, backed by Supabase for customer/ticket data.

- **Backend**: FastAPI REST + WebSocket (Python, `pydantic-ai`)
- **Frontend**: Supabase-authenticated dashboard, live agent console, and customer chat (vanilla HTML/JS/CSS)
- **Memory**: [Hindsight Cloud](https://hindsight.vectorize.io) (per-customer memory banks with TEMPR retrieval)
- **LLM**: Groq (`qwen/qwen3-32b`) via `pydantic-ai` — one call produces response + frustration score + escalation flag + suggested solution + memory summary
- **Database**: Supabase (or in-memory mock mode for development)

## How It Works

### AI Reply Flow (customer sends a message, no agent)

```
Customer POST /my/chat {message}
    │
    ├─ 1. Hindsight recall → relevant memories
    ├─ 2. Hindsight reflect → synthesised context
    ├─ 3. Groq (single call) → response + frustration + escalation + solution
    ├─ 4. Save conversation to Hindsight
    └─ 5. Return {reply, suggested_solution}
```

The LLM receives a layered prompt: customer profile → ticket history → retrieved memories → compact memory summary → Hindsight reflection → customer message. The `_contract_memories()` layer deduplicates memories (collapsing near-duplicates from customer name ×15, email ×20, etc.) and auto-truncates to 8 entries if the prompt exceeds ~5800 estimated tokens — preventing Groq's free-tier 6k limit errors.

### Live Agent Flow (customer + agent connected)

```
Customer POST /my/chat {message}
    │
    ├─ 1-4. Same AI pipeline as above
    ├─ 5. Push memory.update + chat.reply + suggested_reply to agent WebSocket
    └─ 6. Return {reply: "An agent is reviewing…", agent_connected: true}

Agent sees customer message + AI suggestion in real-time
    │
    ├─ Accept suggestion → sendAgentReplyText() → WS {"type":"agent_reply"}
    ├─ Type custom reply  → wireSend()         → WS {"type":"agent_reply"}
    │
    ├─ Backend stores reply as pending, echoes agent_reply_sent
    │   (optimistic render on the agent page avoids duplicate display)
    └─ Customer polls GET /my/pending-replies → receives reply

Agent disconnect → polling continues (agent_disconnected flag)
Agent reconnect → orphaned pending replies flushed as agent_reply_sent
```

### Memory Visualization Pipeline

Every customer message triggers a full memory retrieval timeline that gets pushed to the agent's WebSocket as `memory.update`. The Memory Inspector (at `/memory_inspector.html`) renders the recall graph — query → Hindsight TEMPR retrieval → matched entries with metadata — so agents can see *why* the AI suggested what it did.

### Key Request/Response Schemas

**`POST /my/chat`**
```
Request:  { "message": "I can't log in", "ticket_id": "tkt_001" }
Response: {
  "reply": "I see you've been having login issues…",
  "reply_id": "uuid",
  "suggested_solution": "Reset your password at…",
  "agent_connected": false
}
```

When `agent_connected: true`, the `reply` field contains either the agent's pending reply or `"An agent is reviewing your message."`. The actual AI response is pushed to the agent WebSocket as a `suggested_reply`.

**`GET /my/pending-replies`**
```
Response: {
  "replies": [{"reply_id": "uuid", "text": "Let me check…", "timestamp": "2026-06-15T…"}],
  "agent_disconnected": false
}
```

### WebSocket Events (`/ws/session/{customer_id}`)

| Direction | Event | Payload |
|-----------|-------|---------|
| Server → Agent | `opening` | `{data: "Agent briefing text…"}` |
| Server → Agent | `status` | `{query: "customer message…"}` |
| Server → Agent | `memory.update` | Memory retrieval visualization payload |
| Server → Agent | `chat.reply` | `{data, suggested_reply, suggested_solution, escalation_flag, frustration_score, memory_summary}` |
| Server → Agent | `agent_reply_sent` | `{data: "reply text", reply_id: "uuid"}` (reconnect flush only) |
| Server → Agent | `error` | Error message |
| Agent → Server | `{"type":"agent_reply","text":"…"}` | Agent sends a reply |
| Agent → Server | `{"type":"customer","text":"…"}` | Agent triggers AI for customer |
| Agent → Server | `{"type":"ping"}` | Keepalive (every 25s) |

## Key Components

### `backend/agent.py` — AI Orchestration

The core intelligence layer. One `pydantic-ai` Agent with `GroqModel("qwen/qwen3-32b")` produces all outputs in a single structured call:

- **`SupportResult`**: `response` (str), `frustration_score` (0-100), `frustration_label`, `escalation_flag`, `escalation_reason`, `suggested_solution`, `memory_summary`
- **`_contract_memories()`**: Deduplicates memories by normalizing text, stripping boilerplate (customer name/email/case-number patterns), collapsing near-duplicates via text match, limiting to 12 entries, and building a compact ISSUES / STATUS summary instead of dumping all entries verbatim
- **`generate_support_response()`**: Builds layered prompt → runs agent → clamps frustration → logs everything. Auto-truncates memories if prompt exceeds ~5800 estimated tokens (Groq free-tier limit)
- **`_fallback_result()`**: Keyword-heuristic regex fallback — frustration detection (angry/frustrated/terrible), escalation triggers (lawsuit/legal/security/manager), solution suggestions by intent — ensures the demo never crashes

### `backend/memory.py` — Hindsight Cloud

- One Hindsight bank per customer (bank_id = customer_id)
- Disposition: empathy=5, skepticism=2, literalism=2
- **`retrieve_memories()`**: TEMPR recall → relevance-scored memory entries
- **`reflect()`**: Synthesised natural-language reasoning over all stored memories (budget=high)
- **`save_memory()`**: Retain conversations as experiences
- **`build_retrieval_viz()`**: Structured payload for the frontend Memory Inspector

### `backend/main.py` — Server

- FastAPI lifespan validates env vars and initializes mock/production mode
- CORS configured for `FRONTEND_URL` (default `http://localhost:5173`)
- Service classes: CustomerService, TicketService, MemoryService, LLMService
- SessionManager: in-memory tracking of agent WebSocket sessions + pending reply queues

### Frontend — Key Behaviors

- **Customer polling never stops**: `startPendingPoll()` keeps polling even when agent disconnects; on reconnect the next poll detects `agent_disconnected: false` and resumes
- **Keepalive ping**: Agent WebSocket sends `{"type":"ping"}` every 25s to prevent proxy/load-balancer idle timeouts
- **Reconnect resilience**: Orphaned pending replies are flushed as `agent_reply_sent` on reconnect; frontend deduplicates by tracking `_seenReplyIds`
- **Dedup contract**: Agent replies are rendered optimistically on send; the backend does not echo `agent_reply_sent` on the normal path — only on reconnect flush — so the agent page never shows duplicates

## Setup

### Prerequisites

- Python 3.11+
- A Hindsight Cloud account (for memory) — optional in mock mode
- A Groq API key (free tier at groq.com) — optional; without it, keyword fallback still works

### Environment Variables

Create a `.env` in the project root:

```env
GROQ_API_KEY=gsk_your_groq_key
HINDSIGHT_API_KEY=your_hindsight_key
SUPABASE_URL=https://your_project.supabase.co
SUPABASE_KEY=your_supabase_service_key
FRONTEND_URL=http://localhost:5173
PORT=8000
HINDSIGHT_BASE_URL=https://api.hindsight.vectorize.io
```

### Running

```bash
pip install -r requirements.txt

# Mock mode (no credentials needed for customer/ticket data)
$env:USE_MOCK_DATA="true"
uvicorn backend.main:app --reload --port 8000

# Production mode (requires Supabase credentials)
uvicorn backend.main:app --reload --port 8000
```

### Frontend

Frontend files are static HTML/JS/CSS. Serve them with any HTTP server:

```bash
npx serve frontend/ -l 5173
```

Or open the HTML files directly in a browser (some features require the server for CORS).

### WebSocket Chat Demo (agent console)

Open `http://localhost:5173/liveagent.html` with `?customer_id=cust_001` to connect as an agent. You'll see the opening briefing, memory retrieval, and AI suggestions in real time.

## Development Notes

- **LLM is optional**: Without `GROQ_API_KEY`, the keyword-heuristic fallback (`_fallback_result()`) handles all customer queries. The demo degrades gracefully.
- **`llm.py` was removed**: Its functionality was absorbed into `agent.py`. A compatibility wrapper `generate_response()` exists at `agent.py:398`.
- **Memory dedup is critical**: Without `_contract_memories()`, the prompt can exceed Groq's 6k-token free-tier limit when a customer has 99+ near-duplicate memories (name ×15, email ×20, case-ID variants ×30).
- **Duplicate detection is built in**: Every reply carries a `reply_id` (UUID). The frontend tracks seen IDs in `_seenReplyIds` to prevent double-rendering from poll+WS overlap.
- **No tests yet**: Test framework not configured.
- **Supabase anon key**: Hardcoded in frontend JS files — public by Supabase design, but manageable via env vars.
